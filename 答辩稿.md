# Java 通讯录管理系统 — 答辩稿

## 一、开场（约1分钟）

各位老师好，我是 XX，我的 Java 课程设计题目是**通讯录管理系统**。

这个系统实现了一个完整的桌面端通讯录管理工具，包含**用户注册登录、联系人增删改查、分组管理、多条件模糊查询**五大功能模块。整个项目用 Java Swing 做图形界面，用文本文件做数据持久化，没有使用外部数据库，是一个轻量级、零依赖的桌面应用。

下面我从架构设计、核心功能、关键技术和遇到的问题四个方面来介绍。

## 二、项目架构（约2分钟）

### 2.1 三层分层架构

整个项目分为三层，各层职责清晰，单向调用：

```
界面层（LoginFrame / MainFrame）
    ↓ 调用
业务逻辑层（AddressBookService）
    ↓ 调用
数据层（FileUtil + txt文件）
```

**界面层**负责和用户交互，只做两件事：收集用户输入、把结果展示出来。**业务逻辑层**负责所有业务规则：登录验证、数据校验、增删改查的逻辑处理。**数据层**负责文件的读写，业务层不需要关心文件怎么读、怎么写。

这样分层的好处是：改界面不影响业务逻辑，换存储方式也不影响界面。比如将来想把 txt 文件换成数据库，只需要改 FileUtil 和 AddressBookService 里涉及文件读写的部分，界面层完全不用动。

### 2.2 数据存储方案

本项目选择用 **txt 文本文件** 做数据持久化，存储三个数据文件：

| 文件 | 内容 | 每行格式 |
|------|------|----------|
| `user.txt` | 用户账号密码 | `ID,用户名,密码` |
| `group.txt` | 分组信息 | `ID,分组名,是否默认` |
| `contact.txt` | 联系人信息 | `ID,姓名,电话,省,市,区县,单位,邮箱,备注,分组ID,用户ID` |

选这个方案有三个考量：

**第一，降低部署门槛。** 数据库需要安装、配置、建表，对运行环境有要求。txt 文件随程序走，拷贝过去就能跑，适合作为课程设计的交付物。

**第二，符合课程知识点覆盖要求。** 本项目综合运用了 Java 文件 IO（BufferedReader、PrintWriter）、字符串处理（split、concat）、集合框架（ArrayList、HashMap）等核心知识点，这些在数据库方案里会被框架屏蔽掉，反而不利于展示对 Java 基础的掌握。

**第三，数据量场景匹配。** 通讯录是个人工具，数据量在百条级别，文本文件完全够用，不需要为一个小工具引入数据库这样的重量级组件。

当然，这个方案也有局限性——并发写入不安全、没有事务支持——这些我在总结部分会提到改进方向。

## 三、核心功能详解（约5分钟）

### 3.1 用户注册与登录

**注册流程**在 `AddressBookService.register()` 中实现：

1. 读取 `user.txt` 全部内容，逐行比对用户名是否已存在。
2. 不存在则生成新 ID（当前行数 + 1），追加写入文件。
3. `LoginFrame` 中做了四层校验：非空校验、用户名不少于3位、密码不少于6位、两次密码输入一致。

**登录流程**在 `AddressBookService.login()` 中实现：

1. 读取 `user.txt`，逐行拆分比对用户名和密码。
2. 匹配成功后，将当前用户的 ID 记录到 `loginUserId` 成员变量中。
3. 登录成功后，`LoginFrame` 把 `AddressBookService` 对象传递给 `MainFrame`，复用同一个实例。

**为什么复用 Service 而不是新建？** 因为 `loginUserId` 存在 Service 对象里，如果新建一个 Service，这个 ID 就丢失了，后续查询联系人时无法判断"这是谁的联系人"，会导致数据隔离失效。

### 3.2 联系人增删改查

**新增**（`addContact`）：校验 `loginUserId` 非空（防未登录调用），生成新 ID，把各字段拼接成逗号分隔的字符串，追加到 `contact.txt`。

**查询**（`queryAllContact`）：读取 `contact.txt` 全部数据，逐行解析，通过 `userId != loginUserId` 过滤出当前用户的联系人。同时读取 `group.txt`，用 `HashMap<Integer, String>` 建立分组 ID 到分组名的映射，避免每条联系人都去读一遍分组文件。

**修改**（`updateContact`）：读取全部联系人，遍历时比对 ID，匹配的用新数据替换，不匹配的原样保留，最后覆盖写入文件。

**删除**（`deleteContact`）：使用 Java 8 Stream API，`filter` 过滤掉目标 ID 的行，剩余的收集回列表写入文件。

这四个操作覆盖了文件存储场景下的典型模式：追加写入、遍历比对替换、过滤重写。

### 3.3 分组管理

分组管理的核心逻辑在 `addGroup` 和 `delGroup` 中。

**新增分组**：先检查名称是否重复，不重复则追加写入 `group.txt`。

**删除分组**有一个关键的**数据完整性处理**：删除分组前，必须先把属于该分组的联系人迁移到默认分组。否则这些联系人的分组 ID 指向一个不存在的分组，查询时会出现空分组名。

具体做法是：先找到 `isDefault == 1` 的默认分组 ID，然后遍历 `contact.txt`，把被删分组的联系人的 `groupId` 改为默认分组 ID，再从 `group.txt` 中移除该分组。

默认分组（`isDefault = 1`）做了**保护**，代码中明确判断了 `g.getId() == gid && g.getIsDefault() == 1` 的情况，弹出"默认分组禁止删除"提示并直接返回。

### 3.4 多条件模糊查询

`searchContact` 方法实现了按姓名、电话、省份、单位四个条件的模糊查询。

核心实现用的是 Java 8 Stream API：

```java
return all.stream().filter(c -> {
    boolean match = true;
    if (!name.isEmpty())     match &= c.getName().contains(name);
    if (!phone.isEmpty())    match &= c.getPhone().contains(phone);
    if (!province.isEmpty()) match &= c.getProvince().contains(province);
    if (!company.isEmpty())  match &= c.getCompany().contains(company);
    return match;
}).collect(Collectors.toList());
```

逻辑是：先调用 `queryAllContact()` 拿到当前用户的所有联系人，然后用 `filter` 逐个检查。条件为空的自动跳过，填了的用 `contains` 做包含匹配（即模糊搜索），所有条件之间是"与"关系。

## 四、关键技术点（约2分钟）

### 4.1 面向对象三大特性

**封装**：User、Contact、Group 三个模型类的字段全部用 `private` 修饰，通过 getter/setter 访问。

**继承**：LoginFrame 和 MainFrame 都继承 `JFrame`，复用了窗口的基本能力（标题、大小、关闭操作），各自扩展了登录和主界面的逻辑。

**多态**：JComboBox 的下拉框渲染器通过自定义 `DefaultListCellRenderer`，让分组下拉框显示"分组名(ID)"的格式，而不是默认的 `Group@hashcode`。

### 4.2 集合框架

- `ArrayList`：存储联系人列表、分组列表、文件行数据。
- `HashMap`：在 `queryAllContact` 中，将分组 ID → 分组名建立映射，查询复杂度从 O(n) 降到 O(1)。

### 4.3 文件 IO

- `BufferedReader` + `FileReader`：逐行读取文件。
- `PrintWriter` + `FileWriter`：支持覆盖写入和追加写入两种模式。
- `try-with-resources` 语法：确保文件流在使用后自动关闭，避免资源泄漏。

### 4.4 异常处理

文件操作中可能遇到的异常：
- `IOException`：文件不存在、读写失败。
- `NumberFormatException`：`Integer.parseInt` 解析非数字字符串时抛出。

代码中对这两种异常都做了 `try-catch` 处理，`queryAllContact` 中对脏数据（字段不足、ID 非数字）做了 `continue` 跳过，保证程序不会因为一条脏数据就崩溃。

### 4.5 Swing GUI

- `GridLayout`：登录界面用 4x2 网格布局，组件整齐排列。
- `BorderLayout`：主界面用边界布局，按钮在上、表格在中。
- `JOptionPane`：统一使用对话框进行交互（提示、确认、输入），代码简洁。
- `JTable` + `DefaultTableModel`：表格组件，支持动态增删行。

## 五、遇到的问题与解决方案（约1分钟）

### 问题1：用户数据隔离

**现象**：所有用户看到的是同一份联系人列表。

**解决**：在 `contact.txt` 中增加 `userId` 字段（第11列），`queryAllContact` 查询时通过 `uid != loginUserId` 过滤，只返回当前用户的联系人。登录成功后将 `loginUserId` 保存在 Service 对象中，登录窗口将 Service 传递给主窗口时复用同一个实例。

### 问题2：删除分组后联系人丢失

**现象**：删除分组后，属于该分组的联系人分组名显示为空。

**解决**：`delGroup` 方法在删除分组前，先遍历 `contact.txt`，将被删分组的联系人的 `groupId` 改为默认分组的 ID，再删除分组记录。

### 问题3：编辑时分组下拉框回显

**现象**：编辑联系人时，下拉框默认选中第一个分组，而不是当前联系人所属的分组。

**解决**：遍历所有分组，比较分组名称与当前联系人的分组名称，匹配时调用 `setSelectedItem` 选中对应项。

### 问题4：ID 生成与脏数据

**现象**：删除联系人后，新联系人的 ID 可能与已删除的 ID 重复。

**解决**：当前方案采用"行数 + 1"策略，在小数据量场景下够用。同时 `queryAllContact` 中做了字段数量和类型的校验（`arr.length < 11` 和 `try-catch NumberFormatException`），遇到脏数据直接跳过，不会影响整体功能。

## 六、项目总结与改进方向（约30秒）

本项目实现了一个功能完整的通讯录管理系统，综合运用了面向对象、集合框架、文件 IO、Stream API、Swing GUI 等 Java 核心技术。三层架构设计保证了代码的可维护性，数据完整性处理（分组删除迁移、用户隔离）体现了对业务逻辑的完整思考。

**可改进方向**：

1. **存储层升级**：引入 SQLite 或 MySQL，解决并发安全和数据完整性问题。
2. **安全性**：密码目前是明文存储，应改为哈希加盐存储。
3. **ID 策略**：当前"行数 + 1"在删除后可能重复，可改用自增序列或 UUID。
4. **界面美化**：可升级到 JavaFX 或 Web 前端，提升视觉效果。

## 附录：常见提问与参考回答

**Q：为什么不用数据库？**

> 从课程设计的角度，txt 文件存储有几个优势：一是零依赖，运行环境只需要 JDK，不需要额外安装数据库；二是能完整展示 Java 文件 IO 的知识点，包括 BufferedReader、PrintWriter、try-with-resources 等；三是通讯录是个人工具，数据量在百条级别，文本文件完全能胜任。如果是一个多人并发访问的生产系统，我会选择数据库。

**Q：怎么保证用户只能看到自己的联系人？**

> `contact.txt` 中每条记录的最后一个字段是 `userId`，`queryAllContact` 查询时通过 `uid != loginUserId` 过滤，只返回当前用户的联系人。`loginUserId` 在登录成功时赋值，Service 对象在 LoginFrame 和 MainFrame 之间复用，保证 ID 不丢失。

**Q：删除分组时怎么处理关联数据？**

> `delGroup` 方法分三步：第一步找到默认分组的 ID；第二步遍历所有联系人，将被删分组的联系人的 `groupId` 改为默认分组 ID；第三步从 `group.txt` 中移除该分组。这样保证了联系人不会因为分组删除而丢失归属信息。

**Q：多条件查询是怎么实现的？**

> 先调用 `queryAllContact` 获取当前用户的所有联系人，然后用 Java 8 Stream 的 `filter` 方法逐个检查。四个条件中，空条件自动跳过，非空条件用 `contains` 做包含匹配，条件之间是"与"关系——填了的条件都要满足才会返回。

**Q：Service 对象为什么要从 LoginFrame 传给 MainFrame？**

> 因为 `loginUserId`（当前登录用户的 ID）保存在 Service 对象中。如果 MainFrame 创建一个新的 Service 对象，这个 ID 就是 null，查询联系人时无法过滤，会导致所有用户的数据都显示出来。复用同一个 Service 是保证用户数据隔离的关键。

**Q：项目有什么不足？**

> 主要有三点：一是并发写入不安全，多个进程同时操作同一个文件可能导致数据错乱；二是 ID 生成策略简单，删除后可能重复；三是密码明文存储。这些在生产环境中需要用数据库事务、自增主键、哈希加密等方案解决。

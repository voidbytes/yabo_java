## 二、项目架构（约2分钟）

### 2.1 三层分层架构

整个项目分为三层，各层职责清晰，单向调用：

```
界面层（LoginFrame / MainFrame）
    ↓ 调用
业务逻辑层（AddressBookService）
    ↓ 调用
数据层（DBUtil + MySQL）
```

**界面层**负责和用户交互，只做两件事：收集用户输入、把结果展示出来。**业务逻辑层**负责所有业务规则：登录验证、数据校验、增删改查的逻辑处理。**数据层**通过 `DBUtil` 封装数据库连接，业务层不需要关心连接怎么获取、SQL 怎么执行。

这样分层的好处是：改界面不影响业务逻辑，换存储方式也不影响界面。比如将来想把 MySQL 换成 PostgreSQL，只需要改 `DBUtil` 里的连接地址，业务层和界面层完全不用动。

### 2.2 数据存储方案

本项目使用 **MySQL 8** 做数据持久化，数据库名 `yabo_address_book`，包含三张表：

| 表名 | 内容 | 主要字段 |
|------|------|----------|
| `user` | 用户账号密码 | id, username(唯一), password |
| `grp` | 分组信息 | id, group_name, is_default, user_id |
| `contact` | 联系人信息 | id, name, phone, province, city, county, company, email, remark, group_id, user_id |

表之间的关联关系：
- `grp.user_id` → `user.id`（每个分组属于一个用户），级联删除
- `contact.group_id` → `grp.id`（每个联系人属于一个分组），删除时置 NULL
- `contact.user_id` → `user.id`（每个联系人属于一个用户），级联删除

选 MySQL 的三个考量：

**第一，符合课程要求。** 课程要求使用数据库，MySQL 是最常用的关系型数据库，JDBC 是 Java 操作数据库的标准方式，能完整展示数据库编程的知识点。

**第二，数据完整性。** 数据库通过外键约束、级联删除、唯一约束等机制，从底层保证数据的一致性，不需要在业务代码里手动维护。

**第三，自动初始化。** 程序启动时通过 `DBUtil` 的静态初始化块自动建库建表（`CREATE DATABASE IF NOT EXISTS`、`CREATE TABLE IF NOT EXISTS`），用户无需手动执行 SQL 脚本，零配置即可运行。


## 三、核心功能详解（约5分钟）

### 3.1 数据库自动初始化

`DBUtil` 在类加载时通过静态初始化块完成两件事：

1. **建库**：连接 MySQL（不带库名），执行 `CREATE DATABASE IF NOT EXISTS yabo_address_book`
2. **建表**：连接目标库，执行三张表的 `CREATE TABLE IF NOT EXISTS`（含外键约束）

```java
static {
    Class.forName("com.mysql.cj.jdbc.Driver");  // 加载驱动
    init();  // 建库 + 建表
}
```

`IF NOT EXISTS` 保证多次启动不会重复创建，已有的库和表不受影响。

### 3.2 用户注册与登录

**注册流程**在 `AddressBookService.register()` 中实现：

1. 用 `PreparedStatement` 查询用户名是否已存在。
2. 不存在则插入新用户，通过 `Statement.RETURN_GENERATED_KEYS` 获取数据库自增 ID。
3. 注册成功后，调用 `createDefaultGroup(newUserId)` 为新用户自动创建默认分组。
4. `LoginFrame` 中做了四层校验：非空校验、用户名不少于3位、密码不少于6位、两次密码输入一致。

**登录流程**在 `AddressBookService.login()` 中实现：

1. 用 `PreparedStatement` 按用户名查询用户记录。
2. 取出存储的 BCrypt 哈希值，用 `PasswordUtil.verify()`（底层调用 `BCrypt.checkpw`）验证明文密码是否匹配。
3. 匹配成功后，将数据库返回的自增 ID 记录到 `loginUserId` 成员变量中。
4. 登录成功后，`LoginFrame` 把 `AddressBookService` 对象传递给 `MainFrame`，复用同一个实例。

**为什么不能在 SQL 里直接比对密码？** 因为密码是 BCrypt 哈希存储的，SQL 无法执行哈希运算。必须先查出哈希值，再在 Java 代码中用 BCrypt 验证。

**为什么复用 Service 而不是新建？** 因为 `loginUserId` 存在 Service 对象里，如果新建一个 Service，这个 ID 就丢失了，后续查询联系人时无法判断"这是谁的联系人"，会导致数据隔离失效。

### 3.3 联系人增删改查

**新增**（`addContact`）：校验 `loginUserId` 非空（防未登录调用），用 `PreparedStatement` 执行 `INSERT INTO contact`，数据库自动生成主键。

**查询**（`queryAllContact`）：用一条 `LEFT JOIN` SQL 同时查出联系人和分组名，通过 `WHERE c.user_id = ?` 过滤当前用户的联系人。不再需要在 Java 代码里手动拼 HashMap 做分组映射。

**修改**（`updateContact`）：用 `PreparedStatement` 执行 `UPDATE contact SET ... WHERE id=? AND user_id=?`，`WHERE` 条件中同时校验 ID 和用户归属，防止跨用户篡改。

**删除**（`deleteContact`）：用 `PreparedStatement` 执行 `DELETE FROM contact WHERE id = ? AND user_id = ?`，同样校验用户归属。

### 3.4 分组管理

分组管理的核心逻辑在 `addGroup` 和 `delGroup` 中。

**新增分组**：先按 `group_name` 和 `user_id` 查重，不重复则插入，分组归属当前用户。

**删除分组**使用了**事务**保证数据一致性：

```java
conn.setAutoCommit(false);  // 开启事务
// 1. 将被删分组的联系人迁移到默认分组
// 2. 删除分组记录
conn.commit();  // 提交事务
// 如果中间任何一步出错，自动回滚
```

默认分组（`is_default = 1`）做了**保护**，代码中明确判断后弹出"默认分组禁止删除"提示并返回。

### 3.5 多条件模糊查询

`searchContact` 方法实现了按姓名、电话、省份、单位四个条件的模糊查询。

核心实现用**动态 SQL 拼接**：

```java
StringBuilder sql = new StringBuilder("SELECT ... WHERE c.user_id = ?");
List<Object> params = new ArrayList<>();
params.add(loginUserId);
if (!name.isEmpty()) { sql.append(" AND c.name LIKE ?"); params.add("%" + name + "%"); }
if (!phone.isEmpty()) { sql.append(" AND c.phone LIKE ?"); params.add("%" + phone + "%"); }
// ...
```

逻辑是：根据用户填入的条件动态拼接 SQL，空条件自动跳过，非空条件用 `LIKE` 做模糊匹配，所有条件之间是"与"关系。通过 `PreparedStatement` 的参数化查询防止 SQL 注入。

## 四、关键技术点（约2分钟）

### 4.1 面向对象三大特性

**封装**：User、Contact、Group 三个模型类的字段全部用 `private` 修饰，通过 getter/setter 访问。

**继承**：LoginFrame 和 MainFrame 都继承 `JFrame`，复用了窗口的基本能力（标题、大小、关闭操作），各自扩展了登录和主界面的逻辑。

**多态**：JComboBox 的下拉框渲染器通过自定义 `DefaultListCellRenderer`，让分组下拉框显示"分组名(ID)"的格式，而不是默认的 `Group@hashcode`。

### 4.2 JDBC 数据库编程

- `DriverManager.getConnection()`：获取数据库连接。
- `PreparedStatement`：预编译 SQL，防止 SQL 注入，支持参数化查询。
- `ResultSet`：遍历查询结果。
- `Statement.RETURN_GENERATED_KEYS`：获取自增主键。
- `try-with-resources`：自动关闭 Connection、Statement、ResultSet，避免资源泄漏。

### 4.3 事务处理

`delGroup` 方法中使用了 JDBC 事务：
- `conn.setAutoCommit(false)` 开启事务
- 操作成功后 `conn.commit()` 提交
- 异常时 `conn.rollback()` 回滚

保证"联系人迁移"和"分组删除"两步操作要么全部成功，要么全部回滚，不会出现中间状态。

### 4.4 BCrypt 密码哈希

密码存储使用 **BCrypt** 算法（与 Spring Security 同款），通过 `jbcrypt` 库实现：

- **注册时**：`PasswordUtil.hash(pwd)` → 内部调用 `BCrypt.hashpw(明文, BCrypt.gensalt(10))`，自动生成随机盐并哈希，结果形如 `$2a$10$N9qo8uLOickgx2ZMRZoMye...`（固定60字符）。
- **登录时**：先按用户名查出哈希值，再用 `BCrypt.checkpw(明文, 哈希值)` 验证。BCrypt 从哈希值中提取盐值，用相同盐值哈希明文后比较。
- **安全性**：即使两个用户密码相同，哈希值也不同（因为盐值随机）；彩虹表攻击无效；工作因子为10，暴力破解成本极高。

### 4.4 Swing GUI

- `GridLayout`：登录界面用 4x2 网格布局，组件整齐排列。
- `BorderLayout`：主界面用边界布局，按钮在上、表格在中。
- `JOptionPane`：统一使用对话框进行交互（提示、确认、输入），代码简洁。
- `JTable` + `DefaultTableModel`：表格组件，支持动态增删行。

## 五、遇到的问题与解决方案（约1分钟）

### 问题1：用户数据隔离

**现象**：所有用户看到的是同一份联系人列表。

**解决**：`grp` 表和 `contact` 表都有 `user_id` 字段，SQL 查询时通过 `WHERE user_id = ?` 过滤，只返回当前用户的数据。分组和联系人都是按用户隔离的。

### 问题2：删除分组后联系人丢失

**现象**：删除分组后，属于该分组的联系人分组名显示为空。

**解决**：`delGroup` 方法使用事务，先将被删分组的联系人的 `group_id` 改为默认分组的 ID，再删除分组记录。外键约束 `ON DELETE SET NULL` 作为兜底保护。

### 问题3：数据库自动初始化

**现象**：首次运行时数据库和表不存在，程序无法启动。

**解决**：`DBUtil` 的静态初始化块分两步执行：先连接 MySQL 不带库名创建数据库，再连接目标库创建表。`IF NOT EXISTS` 保证多次启动无副作用。

### 问题4：编辑时分组下拉框回显

**现象**：编辑联系人时，下拉框默认选中第一个分组，而不是当前联系人所属的分组。

**解决**：遍历所有分组，比较分组名称与当前联系人的分组名称，匹配时调用 `setSelectedItem` 选中对应项。

## 附录：常见提问与参考回答

**Q：为什么用 MySQL 而不是 txt 文件？**

> MySQL 是关系型数据库，通过外键约束保证数据完整性，通过事务保证操作的原子性，通过自增主键避免 ID 冲突。JDBC 是 Java 操作数据库的标准 API，掌握 JDBC 是后端开发的基本功。

**Q：数据库是怎么自动初始化的？**

> `DBUtil` 类有一个静态初始化块，在类第一次被加载时自动执行。它先连接 MySQL 服务器（不指定数据库），用 `CREATE DATABASE IF NOT EXISTS` 创建数据库；再连接目标数据库，用 `CREATE TABLE IF NOT EXISTS` 创建三张表。所有 SQL 都用了 `IF NOT EXISTS`，多次运行不会重复创建。

**Q：怎么保证用户只能看到自己的联系人？**

> `contact` 表有 `user_id` 字段，所有查询 SQL 都带 `WHERE user_id = ?` 条件，`?` 的值是登录成功后保存在 `loginUserId` 中的当前用户 ID。分组表同理，也有 `user_id` 字段做隔离。

**Q：删除分组时怎么处理关联数据？**

> `delGroup` 使用 JDBC 事务，分两步：第一步用 `UPDATE` 将被删分组的联系人迁移到默认分组；第二步用 `DELETE` 删除分组记录。两步操作在同一事务中，要么全部成功，要么全部回滚。此外，外键约束 `ON DELETE SET NULL` 作为兜底，即使漏删也能保证数据不丢失。

**Q：多条件查询是怎么实现的？**

> 根据用户填入的条件动态拼接 SQL 的 `WHERE` 子句，空条件跳过，非空条件用 `LIKE ?` 做模糊匹配。所有参数通过 `PreparedStatement` 的 `setObject` 设置，既防止 SQL 注入，又支持灵活的条件组合。

**Q：Service 对象为什么要从 LoginFrame 传给 MainFrame？**

> 因为 `loginUserId`（当前登录用户的 ID）保存在 Service 对象中。如果 MainFrame 创建一个新的 Service 对象，这个 ID 就是 null，查询联系人时无法过滤，会导致所有用户的数据都显示出来。复用同一个 Service 是保证用户数据隔离的关键。

**Q：项目有什么不足？**

> 主要有两点：一是每次操作都新建数据库连接，高并发场景下应该用连接池；二是没有日志系统，排查问题只能看控制台输出。这些在生产环境中需要对应的技术方案来解决。

**Q：密码是怎么存储的？为什么不在数据库里直接存明文？**

> 密码使用 BCrypt 算法哈希后存储。注册时调用 `BCrypt.hashpw(明文, BCrypt.gensalt(10))`，自动生成随机盐并哈希，结果形如 `$2a$10$N9qo8uLOickgx2ZMRZoMye...`（60字符）。登录时先按用户名查出哈希值，再用 `BCrypt.checkpw(明文, 哈希值)` 验证。即使两个用户密码相同，哈希值也不同（盐值随机），彩虹表攻击无效，暴力破解成本极高。
